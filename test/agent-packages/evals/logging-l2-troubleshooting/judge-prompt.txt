# Rubric

Each check is binary (pass or fail). Do not give partial credit.

Strict reading rules:
- Mark FAIL whenever the rubric requires explicit evidence and the
  transcript does not contain that evidence verbatim.
- Do not infer intent. If a rubric check asks for a specific YAML field
  with a specific value, the literal field-and-value must appear in the
  transcript text. Synonyms and paraphrases do not satisfy the check.
- A check that says "vacuous pass if X" must pass when X holds, with
  no further evidence required.

# Transcript

The transcript is a JSON document with three top-level keys:

- `result` — the agent's final assistant message (string). Usually
  contains the closing `recommend:` block or a manual-diagnosis
  statement.

- `assistantMessages` — array of all assistant text messages from the
  session, in chronological order. Intermediate `hypothesis_refuted:`
  and `recommend:` YAML blocks emitted by area skills mid-chain live
  here, NOT in `result`. When a rubric check asks about an
  intermediate refute or a per-hop output, search this array.

- `toolCalls` — array of tool invocations in chronological order, each
  with `name`, `input`, `output`, and `is_error`. Authoritative for
  chain ordering. Conventions:
    - `name == "Skill"` with `input.skill == "<id>"` = agent invoked
      skill `<id>`. Order in `toolCalls` is the order of invocation.
    - `name == "Bash"` with `input.command` containing kubectl / helm
      / curl = a shell action; `output` carries its result.
    - Any non-list-style verb (`apply`, `edit`, `delete`, `patch`,
      `scale`, `rollout restart`, `helm upgrade`, `helm rollback`,
      `curl -X POST/PUT/DELETE` against an API) = a mutating call.

To locate a specific area skill's output: find its Skill call in
`toolCalls`, then read the assistantMessages emitted between that call
and the next Skill call (or, for the last invocation, between the call
and the final `result`).

Refute / recommend YAML schemas (canonical shapes the experts emit):

  hypothesis_refuted: true
  skill: <skill-name>
  signal_class: clean | secondary_backpressure | secondary_quoted
  diagnostic_pass_evidence: |
    ...
  cited_external_components:
    - ...
  reason: ...

  recommend:
    what: ...
    why: ...
    blast_radius: ...
    rollback: ...
    snapshot: ...
    confidence: ...

A `hypothesis_refuted: true` block and a `recommend:` block are
mutually exclusive in a single area skill's turn. A refute means the
chain advances; a recommend means the case is closed at that skill.

# Output

Return strict JSON. No prose outside the JSON. Schema:

{
  "pass": true | false,
  "score": <number between 0 and 1, equal to passing_checks / total_checks>,
  "reason": "<one short line summarising overall_pass and pass count, e.g. 'overall_pass=false (3/6 checks passed)'>",
  "checks": [
    {
      "id": "<rubric check id>",
      "pass": true | false,
      "evidence": "<one short verbatim quote from the transcript, or '' if pass=false>"
    }
  ]
}

`pass` is true if and only if every entry in `checks` has `pass=true`.
`score` is the fraction of checks that passed, between 0.0 and 1.0.
The order of entries in `checks` MUST match the order in the rubric above.
